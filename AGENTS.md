# Repository Guidelines

## 项目结构与模块组织
- `docs/` 是当前项目的事实来源：设计说明、协议规范与规划文档。
  - `docs/protocol-spec.md` 定义 HTTP/JSON API、资源模型与错误码。
  - `docs/remote-printing-design.md` 与 `docs/Remote_Printing_System_Development_Documentation.md` 描述系统架构及 CLI/GUI 路线图。
- 目前仓库尚无应用源码；请将其视为“文档优先”的工作区。

## 构建、测试与开发命令
- 目前仓库未提交任何构建或测试工具。
- 若新增代码，请在此处补充实际命令（例如：`cargo build`、`cargo test`、`npm run dev`），并与脚本保持一致。

## 编码风格与命名规范
- 文档：使用 Markdown，结构清晰，段落短。
- 文件命名：使用中文字符（例如：`端口规范.md`）。
- 示例要具体且限定于本项目范围（例如：`/jobs/{id}/content`）。

## 测试指南
- 目前未配置测试框架。
- 进入实现阶段后，建议为协议处理与作业生命周期状态转换添加单元测试，并在此说明运行方式。

## 提交与合并请求指南
- 当前目录没有可参考的提交历史或约定。
- 若初始化版本控制，请使用简短、祈使式的提交标题，行为变更请在正文补充上下文。
- 合并请求应包含：清晰说明、关联问题或文档链接，以及显式标注任何 API 或协议变更。

## 安全与配置提示
- 不要在仓库内保存敏感信息（令牌、认证密钥、证书）；改用环境变量或本地配置。
- 记录认证流程时，示例保持通用（例如：`Authorization: Bearer <token>`）。
